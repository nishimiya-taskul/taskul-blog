# TASKUL Blog - SEO記事制作ルールブック

## プロジェクト概要
- **サイト**: taskul-ai.com/column/
- **技術スタック**: Next.js + Tailwind CSS + Markdown (SSG)
- **記事フォーマット**: `posts/` ディレクトリにMarkdownファイルを配置
- **メインターゲット**: フリーランス・クリエイター（Webデザイナー、イラストレーター、動画クリエイター、ライター、Webマーケター等）
- **サブターゲット**: 制作会社、広告代理店（案件管理・チームタスク管理の文脈で接点を作る。取れたら熱い層）
- **目的**: SEOで集客 → TASKUL への無料登録CV

### ブログ記事の公開フロー（この手順を厳守すること）

```
Step 1: taskul-blog で記事を書く
  → ~/Desktop/Github/taskul-blog/posts/ に記事Markdownを作成

Step 2: taskul-blog でブランチ作成 → コミット → push → PR作成 → マージ
  → マージ → Vercelが自動ビルド＆デプロイ → taskul-ai.com/column/ に反映
```

※ ローカルビルドやtaskul-lpへのコピーは不要。Vercelが自動でやる。

### 絶対にやってはいけないこと
- `taskul`（アプリ）リポジトリにブログ関連の変更を入れる
- 「とりあえずマージして確認」を繰り返す（1デプロイ = コスト）
- デプロイ先を確認せずにPRをマージする

## 記事制作フロー（必ずこの順番で実行する）

### Phase 1: リサーチ（記事を書く前に必ずやる）

#### 1-1. キーワード調査
- WebSearchで「ターゲットKW」を検索し、上位10記事のURL・タイトル・見出し構成を全て取得する
- サジェストキーワードを調査する（「KW あ」「KW か」「KW さ」...で検索）
- 「他の人はこちらも質問」「関連する質問」を全て記録する
- 検索ボリュームの推定（上位記事の競合度から判断）

#### 1-2. 競合分析（最重要）
上位5記事について以下を分析する:

```
各記事ごとに:
- URL
- タイトル
- H2/H3の見出し一覧
- 記事の文字数（概算）
- 独自の情報や図解があるか
- CTAの種類と位置
- 記事の弱点（情報が古い、具体性がない、特定の視点が欠けている等）
```

#### 1-3. 検索意図の特定
検索者を以下の4タイプに分類し、ターゲットKWがどれに該当するか判断する:

| タイプ | 検索意図 | 例 | 記事の方向性 |
|--------|---------|-----|------------|
| Know | 情報を知りたい | 「タスク管理とは」 | 解説・教育型 |
| Do | やり方を知りたい | 「タスク管理 始め方」 | ハウツー型 |
| Compare | 比較して選びたい | 「タスク管理ツール 比較」 | 比較表型 |
| Solve | 悩みを解決したい | 「タスク管理 続かない」 | 課題解決型 |

### Phase 2: 構成設計（書く前に構成を確定する）

#### 2-1. 見出し設計ルール
- **上位5記事中4記事以上にある見出しテーマ** → 必ず含める（Googleの期待値）
- **上位記事にないが検索者が知りたいはずの情報** → 差別化として追加
- **検索者の心理フローに沿った順番**にする:

```
悩みの共感（なぜこの記事を読むべきか）
  → 原因の深掘り（なぜうまくいかないのか）
    → 解決策の提示（どうすればいいのか）
      → 具体的な方法・ツール紹介（何を使えばいいか）
        → すぐできるアクション（今日から何をすればいいか）
```

#### 2-2. タイトル設計ルール
- 主要KWを**タイトルの前半**に入れる
- 32文字以内を目安（Google表示幅）
- 以下のいずれかのフォーマットを使う:
  - 「〇〇が△△な理由と、具体的な解決策」（課題解決型）
  - 「〇〇の□□術｜△△を一元管理する方法」（ハウツー型）
  - 「〇〇と感じたら読む記事｜△△な選択肢」（不満キャッチ型）
- **クリックしたくなる要素**を1つ入れる（数字、ネガティブワード、具体性）

#### 2-3. 構成テンプレート
```markdown
## [検索者の悩みに共感するH2]（冒頭で離脱を防ぐ）

### [原因1]
### [原因2]
### [原因3]

## [解決策・考え方を提示するH2]

### [ポイント1]
### [ポイント2]
### [ポイント3]

## [具体的なツール・方法を比較するH2]（CVに最も近いセクション）

### [ツールA]
### [ツールB]
### [TASKUL]（最後に配置。比較の流れで自然に訴求）

## [比較まとめ表]（あれば）

## [今日からできるアクション]（行動喚起）

## まとめ
```

### Phase 3: 執筆ルール

#### 3-1. 文章のルール
- **1文は60文字以内**を目安にする
- 漢字比率30%以下（読みやすさのため）
- 専門用語は初出時に必ず説明する
- 「です・ます」調で統一
- 「〜と思います」「〜かもしれません」は使わない（断言する）
- 冒頭200文字以内に主要KWを含める

#### 3-2. SEO技術ルール
- **H1**: 記事タイトル（1記事に1つだけ。frontmatterのtitle）
- **H2**: 主要KWまたは関連KWを自然に含める
- **H3**: H2を具体化する小見出し。KWの詰め込みはしない
- **H4以下**: 必要な場合のみ使用
- 見出しの階層を飛ばさない（H2→H4はNG、H2→H3→H4はOK）
- 画像にはalt属性を必ず設定する（KWを含む説明文）

#### 3-3. TASKUL訴求のルール（最重要）
- **記事の50%以上は純粋に有益な情報**にする（宣伝記事にしない）
- ツール比較セクションでは**最低3つのツール**をフェアに紹介する
- TASKULは比較の最後に配置する（「一番良い」ではなく「こういう人に合う」）
- 無料で試せることを必ず伝える
- **他ツールをディスらない**（「Notionは悪い」ではなく「Notionは高機能すぎる」）
- CTAは記事中盤と末尾の2箇所まで（入れすぎない）

#### 3-4. 独自性の担保（上位記事との差別化）
以下のうち最低2つを記事に含める:
- **具体的な数字**（「平均47秒→3秒に短縮」等）
- **比較表**（他記事にないオリジナルの比較軸）
- **チェックリスト**（読者がすぐ使えるもの）
- **ステップバイステップ**（「今日からできる3ステップ」等）
- **あるある・失敗パターン**（検索者の共感を得る実例）

#### 3-5. 内部リンクルール
- 関連する既存記事があれば、本文中に自然にリンクを入れる
- リンクテキストにはKWを含める（「詳しくはこちら」はNG）
- 1記事あたり2〜4本の内部リンクが目安

### Phase 4: 品質チェック（公開前に必ず実行）

記事を書き終えたら、以下のチェックリストを実行する:

```
□ タイトルに主要KWが含まれているか
□ H2に関連KWが自然に含まれているか
□ 冒頭200文字以内に主要KWがあるか
□ 見出し階層が正しいか（H1→H2→H3の順）
□ 比較セクションで3ツール以上紹介しているか
□ TASKULの訴求が押し売りになっていないか
□ 記事の50%以上が純粋に有益な情報か
□ 上位記事にある「必須テーマ」を網羅しているか
□ 上位記事にない「差別化テーマ」が1つ以上あるか
□ 具体的な数字・比較表・チェックリストのいずれかがあるか
□ 内部リンクが2本以上入っているか
□ description（meta）が120文字以内で魅力的か
□ 画像のalt属性が設定されているか
```

### Phase 5: 公開後の改修サイクル

#### 5-1. 計測スケジュール
| タイミング | やること |
|-----------|---------|
| 公開直後 | GSCでインデックスリクエスト |
| 2週間後 | インデックスされているか確認 |
| 1ヶ月後 | 順位・表示回数・CTRを確認 |
| 2ヶ月後 | 本格的なリライト判断 |
| 以降毎月 | 全記事の順位チェック→リライト |

#### 5-2. リライト判断基準
| 状態 | 対応 |
|------|------|
| 表示回数多い + CTR低い（2%以下） | **タイトルとdescriptionを変更** |
| CTR高い + 順位低い（11〜20位） | **記事の中身を強化**（情報追加、独自性UP） |
| 順位高い（1〜5位）+ CV低い | **CTA周りを改善**（位置、文言、デザイン） |
| 圏外（30位以下） | **構成から作り直し**（検索意図を再分析） |
| 順位が下がってきた | **競合の新着記事を分析**→足りない情報を追加 |

#### 5-3. リライトルール
- リライト時は**既存のURLを変えない**（SEO評価を引き継ぐ）
- 日付を更新する（frontmatterのdate）
- 変更内容が大きい場合はtitleも見直す
- リライト前後で順位を記録して効果測定する

## 禁止事項

- 根拠のない数字を入れない（「CVR300%改善」等の裏付けのない数字）
- 競合ツールを直接的に否定しない
- 「絶対」「必ず」「100%」等の誇大表現を使わない
- 記事タイトルに【PR】なしでTASKULを前面に出さない（比較記事の体裁を保つ）
- 同じ記事構成を使い回さない（Googleは類似コンテンツを評価しない）
- postsフォルダ以外のファイル（テンプレート・コンポーネント）を変更しない

## 画像ルール

### 記事作成時のダミー画像挿入（必須）
記事を書く際、以下のタイミングで**必ずダミー画像のプレースホルダーを挿入**する:

- **H2の直後**（各セクションの冒頭にイメージ画像）
- **比較表の前後**（図解やスクリーンショットが入る想定）
- **手順説明**（ステップごとの画面キャプチャ等）
- **記事冒頭**（アイキャッチ/ヒーロー画像）

ダミー画像の書き方:
```markdown
![フリーランスがタスク管理ツールを操作している様子](/images/articles/freelance-task-management/01.png)
<!-- 画像未設定: フリーランスの作業デスクのイメージ写真 or TASKULの画面キャプチャ -->
```

- `![alt文]` にはKWを含む説明文を入れる
- **HTMLコメントで「どんな画像が入るべきか」を必ず書く**（後から画像を用意する人への指示）
- 1記事あたり4〜6枚が目安
- **縦長（ポートレート）画像は禁止** — 必ず横長（ランドスケープ）を使う
- Pexels APIでは `orientation=landscape` を必ず指定する
- ダミー画像は空の画像パスでOK（ファイルがなくてもビルドは通る）

### 画像の配置場所
- 記事内画像: `public/images/articles/{記事slug}/01.jpg, 02.jpg, ...`
- サムネイル: `public/images/articles/{記事slug}/thumbnail.png`
- **TASKULスクショ元データ: `~/Desktop/taskulスクショ/`**（必ずここから取得。リポジトリ内の古いスクショを使わない）

### 画像ファイルの取得・配置フロー
1. **Pexels API**でイメージ写真を取得し `public/images/articles/{slug}/01.jpg` 等に保存
2. **TASKULスクショ**は `~/Desktop/taskulスクショ/` から適切な画像を選び `public/images/articles/{slug}/` にコピー
3. 記事Markdown内で画像パスとalt属性を設定
4. ダミー画像（コメント付き未設定パス）を残さない — 必ず実画像を入れて公開する

## チャットブロック（会話要素）

記事の要所に、人間とTASKUL AIの1往復チャットを挿入する。記事内容を後押しする補足や、読者の悩みを代弁する役割。

### 配置ルール
- 1記事あたり2〜3箇所が目安
- H2セクションの切れ目（次のH2の直前）に配置すると自然
- 人間側は「読者が思っていそうな悩み・疑問」を代弁する
- TASKUL AI側は「記事の次のセクションへの橋渡し」になる一言を返す
- 宣伝臭くしない。あくまで有益なアドバイスとして

### HTMLテンプレート
```html
<div class="chat-block">
<div class="chat-human">
<div class="chat-human-icon"><img src="/images/chat-user.png" alt="ユーザー"></div>
<div class="chat-human-content">
<span class="chat-name">フリーランスデザイナー</span>
<div class="chat-bubble">ユーザーの悩みや質問をここに</div>
</div>
</div>
<div class="chat-ai">
<div class="chat-ai-row">
<div class="chat-ai-icon"><img src="/images/chat-taskul.png" alt="TASKUL AI"></div>
<div class="chat-ai-content">
<span class="chat-name">TASKUL AI</span>
<div class="chat-bubble">TASKUL AIの回答をここに</div>
</div>
</div>
</div>
</div>
```

### デザイン
- **人間（左）**: 女性アイコン + グレー吹き出し（`#eef1f6`）。質問する側
- **TASKUL AI（右）**: 忍者アイコン + 緑吹き出し（`#15cf78`、白文字）。教える側
- chat-nameは記事の内容に応じて変える（「フリーランスデザイナー」「Webマーケター」「イラストレーター」等）
